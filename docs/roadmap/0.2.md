# Roadmap — 0.2

Items deliberately deferred from the 0.1 release. Each entry is "revisit if and when demand appears" rather than a committed milestone. 0.2 planning will pick up whichever of these have accumulated real use cases by the time it starts.

Sources: open questions and out-of-scope notes distilled from the 0.1 specs under `docs/**`.

## IR

- **ANTLR-style cardinality (`{n,m}`)** — would require `minCount` / `maxCount` on `repetition`. `parser-antlr` shipped in 0.1 without it; grammars using this syntax currently raise.

## Renderer

- **Layout debugging mode** — an option to draw bounding boxes / measurement debug overlays. Useful the day we start having layout regressions.
- **Custom text metrics** — text width today is estimated from a fixed character-width heuristic inherited from the legacy project. Real glyph metrics would need a font provider abstraction.
- **Multi-diagram documents** — emitting several diagrams in one SVG (for a full grammar) is currently the consumer's job: call `render` per rule and stack the results. A `renderGrammar(...)` helper could be offered later if there is demand.

## Manual builder

- **Fluent chaining** — a `.then(...)` / `.or(...)` API on top of the factories for a more DSL-like feel. Would live in a separate package to keep `core` lean.
- **Bulk factories** — shortcuts like `terminals("a", "b", "c")` returning an array. Cheap to add later if real usage shows the need.
- **Source-aware builder variant** — a parallel set of factories that accept and attach a `SourceRange`. The spread-then-assign pattern covers today's rare cases, and parsers (the high-volume producer) do not use the builder.

## Bindings

### React (`@choo-choo/react`)

- **Event surface for parse errors** — an `onError` callback to complement the "throw and let the boundary catch it" flow. Waiting on a real use case.
- **`React.memo`-by-default wrapper** — would skip re-renders when props are referentially equal. Consumers can apply it themselves cheaply.

### Vue (`@choo-choo/vue`)

- **Event surface for parse errors** — an `@error` emit to complement the throw-and-catch flow. Waiting on a real use case.
- **Provide / inject for a shared parser** — let a tree of diagrams use one parser instance without re-passing it as a prop.

### Vanilla (`@choo-choo/vanilla`)

- **`ParsedGrammar` adapter on the element** — render every rule of a multi-rule grammar as stacked diagrams. Today users iterate and call `mount()` per rule.
- **`MutationObserver` over children** — for reactive updates when `element.textContent = "..."` is reassigned outside of attribute/property paths. Today the element captures children once on connect and on attribute changes.
- **Event detail surface** — `choo-choo-render` currently sends only `{ svg }`. A resolved `Diagram` could be added if a use case surfaces.
- **Shadow DOM opt-in** — a `shadow="open"` attribute. Today the element uses light DOM so external CSS can style its innards.

### Astro (`@choo-choo/astro`)

- **Per-page error fallback** — consumers wrap in their own `try/catch` today.
- **`<ChooChoo>` as a named export** — the default-export shape matches other Astro integrations.

## Grammar parsers

### EBNF (`@choo-choo/parser-ebnf`)

- **Exception (`-`)** — ISO's `syntactic factor = [integer, "*"], syntactic primary` with the syntactic-exception operator.
- **Integer multiplier (`N *`)** — fixed-count repetition.
- **Empty sequence** — the ISO concept is syntactically absent in our subset; users express optionality with `[…]` instead.
- **Permissive BNF mode** — accept `::=` as a synonym for `=` and `<name>` as nonterminal syntax. A follow-up feature on `parser-ebnf` itself, not a separate parser.
- **Nested comments** — ISO 14977 does not require them; the legacy didn't support them either.
- **`.` as rule terminator** — today only `;` is accepted.

### ANTLR (`@choo-choo/parser-antlr`)

- **Modes as structured regions** — `mode X;` is parsed and consumed, but rules declared under a mode land in `ParsedGrammar.rules` flat, indistinguishable from top-level lexer rules. A later iteration may group them.
- **Grammar-header kind tracking** — `parser grammar`, `lexer grammar`, plain `grammar` are all accepted but the parser does not expose which kind was declared.
- **Import resolution** — `import X, Y;` is consumed but the imported grammars are not fetched or parsed.
- **Character-class expansion** — charsets render as one `special` node with the literal source text, not as a `choice` of individual terminals. This is a deliberate 0.1 choice — a `[a-zA-Z0-9_]` railroad with 63 branches is visually useless — but worth revisiting if there is a clear UX win.
- **Doc comments as `comment` IR nodes** — `/** … */` is currently dropped like any other comment. A follow-up can promote them to `comment` nodes attached to the next rule.
- **`EOF` as a distinct end marker** — treated as a plain `nonterminal("EOF")`.
- **Nested comments** — ANTLR4 itself doesn't support them.

### PEG (`@choo-choo/parser-peg`)

- **Ordered-choice ordering in the diagram** — `/` and `|` both render as unordered vertical branches. Consumers read source order.
- **Case-insensitive semantics** — the `i` flag is preserved textually in the rendered label but not interpreted; we don't case-fold terminals.
- **Typed rules or imports** — peggy's plugin ecosystem is out of scope.
